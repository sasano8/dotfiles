# docs-summary — docs を解析して再整理した要約を作る

`.cache/docs/` に集めたドキュメント（主に `meetings/` の議事録）を読み、**横断的に再整理した要約**を
`.cache/docs/summary/` に生成して表示するスキル。

> このスキルは**置いてあるファイルを解析するだけ**（ソースの作成はしない）。議事録などの新規作成は
> `bin/new-meeting` を使う。

## 対象（docs ベース）

- 既定: **`~/projects/dotfiles/.cache/docs`**（個人ナレッジベース。`.cache/` はローカル限定＝git 非追跡）。
- ユーザーが別の docs パスを指定した場合はそちらを対象にする。
- 解析対象は docs ベース配下の `*.md`（再帰）。ただし **`summary/` 自身は除外**（生成物を読み込まない）。

## 手順

1. **収集** — docs ベース配下の Markdown を列挙（`summary/` を除く）。`meetings/` のファイル名は
   `yyyymmddhhmm.md`（時刻）なので時系列の手がかりに使う。0 件なら「対象ドキュメントがありません」と伝えて終了。
2. **解析** — 各ドキュメントから次を抽出する：
   - 主要トピック / 論点
   - **決定事項**（議事録の「決定事項」節など）
   - **未完了の ToDo**（`- [ ]` のチェックボックス。`- [x]` は完了として除外）と、分かれば担当・期日
   - 日付・参加者などのメタ情報
3. **再整理** — ファイル単位の羅列ではなく、**トピック別に統合**する。重複・言い換えは束ね、決定事項は
   時系列で並べ、未完了 ToDo は 1 か所に集約する。出典（元ファイル名）を各項目に併記する。
4. **生成** — `<docsベース>/summary/SUMMARY.md` を**再生成（上書き）**する（派生ビュー＝source は docs 本体）。
   下記「出力フォーマット」に従う。冒頭に生成元の件数・対象期間を書く。
5. **表示** — 生成した SUMMARY.md の内容を要点とともにユーザーへ提示する。

> `.cache/` 配下はローカル限定（git 非追跡）なので **コミットはしない**。生成物として手元に置くだけ。

## 出力フォーマット（summary/SUMMARY.md）

```markdown
# Docs Summary — 生成: <yyyy-mm-dd hh:mm>

対象: <件数> 件（<最古の日付> 〜 <最新の日付>）

## トピック別まとめ
### <トピック A>
- 要点（出典: meetings/<file>.md）
- ...

## 決定事項（時系列）
- <yyyy-mm-dd> <決定内容>（出典: ...）

## 未完了 ToDo（横断）
- [ ] <内容> — 担当/期日（出典: ...）

## 時系列
- <yyyy-mm-dd hh:mm> <概要>（<file>）
```

## 注意

- `summary/` は**生成物**。手で書いた内容を上書きで失わないよう、SUMMARY.md 以外の手書きファイルが
  summary/ にある場合は上書きせず別名（例 `SUMMARY.generated.md`）にするかユーザーへ確認する。
- 推測で事実を作らない。出典に無い決定・担当・期日は「未記載」とする。
- 大量にある場合はトピックごとに分割ファイル（`summary/<topic>.md`）にしてもよい（その旨を SUMMARY に索引）。
