# Memory Bank（Cline 準拠）

> 本スキルは [Cline の Memory Bank](https://docs.cline.bot/prompting/cline-memory-bank) の構成・運用に準拠する。
> Memory Bank の実体は **作業フォルダ `.work/skills/memory-bank/`** 配下に置く（`.work/skills/<スキル名>/` 規約。
> リポジトリ直下を汚さない）。`.work/` は **gitignore しない**＝引き継ぎのため commit する状態の正本。

私（このエージェント）はセッションを跨ぐと記憶が完全にリセットされる、という前提で動く。リセット後は
**Memory Bank だけ**を頼りにプロジェクトを理解し作業を継続する。したがって **毎タスクの開始時に Memory Bank の
全ファイルを必ず読む（任意ではない）**。

## 0. 起動時チェック（必須・最初に）

`.work/skills/memory-bank/` の有無と 6 コアファイルの存在を確認する。

- **揃っている** → 全ファイルを読み、続けて **`interrupt/`（受信箱ディレクトリ）を確認**する。未処理の
  ファイルがあればコアワークフローに入る前に取り込む（下記「割り込み（interrupt）受信箱」）。
  その後「コアワークフロー」へ。
- **無い／不完全** → ここで**いったん止まる（ブロッキング）**。「Memory Bank が無い／コアが欠けている」と
  **通知するだけで先の作業に進んではいけない**。`initialize memory bank` をユーザーに提案し、**承諾されたら
  実際に作り切る**（下記「初期化」＝`mkdir` ＋ 6 コア雛形の書き出しまで完了させる。通知や提案で止めない）。
  ユーザーが明示的に「不要」と断ったときだけ初期化せず進み、その旨を一言残す。
  **禁止パターン: 「無い／壊れていると通知 → 作成せず次の作業へ」**（成立条件 `.work`＋6 コアが欠けたまま
  作業すると、リセット後の引き継ぎが成立しない＝Memory Bank の意味が無くなる）。

旧レイアウト（リポジトリ直下や `AGENT_LOOP.md` / `PROJECT.md` の 2 ファイル構成）が見つかった場合は、
Cline 6 ファイル構成へ移行することを提案する（内容を 6 ファイルへ振り分け、旧ファイルは削除）。

## Memory Bank の構造（コアファイル）

すべて Markdown。ファイルは階層を成し、上位が下位を規定する。

```
projectbrief.md ─┬─> productContext.md ─┐
                 ├─> systemPatterns.md ─┼─> activeContext.md ─> progress.md
                 └─> techContext.md ────┘
```

### コアファイル（必須・6 つ）

1. **projectbrief.md** — 全ての土台。プロジェクト開始時に作成。**コア要件・ゴール・スコープの正本**。
2. **productContext.md** — なぜ存在するか／解決する課題／どう動くべきか／UX ゴール。
3. **activeContext.md** — **現在の作業フォーカス**／直近の変更／次のステップ／進行中の決定・考慮事項／
   重要なパターンと好み／学び・知見。（最も頻繁に更新する）
4. **systemPatterns.md** — システム構成／主要な技術判断／採用する設計パターン／コンポーネント関係／
   重要な実装経路。
5. **techContext.md** — 使用技術／開発セットアップ／技術的制約／依存／ツールの使い方。
6. **progress.md** — 動くもの／残作業／現状ステータス／既知の問題／意思決定の変遷。

### 追加コンテキスト（任意）

整理に役立つなら `memory-bank/` 配下にファイル/フォルダを足してよい（複雑な機能の仕様、統合仕様、
API ドキュメント、テスト戦略、デプロイ手順 など）。

### 参照系（reference/）

横断的に参照される定義・要件は **`reference/`（ファイルでもディレクトリでもよい）** に集約する。各エントリは
「1 つの要件・参照」を定義し、詳細は所有元（スキル等）へ委ねる。これで**個別要件を 1 か所に集約**でき、
新しい要件を足しやすい。

- 例: `reference/quality-policy.md` … 品質チェックの参照（組織の品質方針 → [[quality]]）。
- **品質チェック以外の要件も同様に追加してよい**（例: リリース手順・セキュリティ要件・命名規約・依存方針 など）。
- コアファイルや SKILL 本体は要件の中身を持たず **`reference/` を参照するだけ**にする（規約の二重持ち＝drift 源を避ける）。
- 要件ごとに 1 ファイルで増やせるよう**ディレクトリ推奨**。1 つだけなら `reference.md` 単体でもよい。

## コアワークフロー

### Plan Mode（計画）

```
Start → Memory Bank を読む → ファイルは揃っているか?
  ├─ No  → 計画を立てる → チャットに計画を文書化
  └─ Yes → コンテキストを検証 → 戦略を立てる → 進め方を提示
```

### Act Mode（実行）

```
Start → Memory Bank を確認 → interrupt を取り込む（作業系の対話入力はまず interrupt へ。問い合わせ/コミット等は除く）
      → ドキュメントを更新 → タスクを実行 → 変更を文書化
      → 切りがよければ commit（下記「コミットをフローに組み込む」）
```

「変更を文書化」したら、**切りのいいところでコミットする**ところまでが 1 サイクル。コミットせずに
次のタスクへ進まない（コミットは引き継ぎの単位＝リセット後の唯一の手がかりを確定させる行為）。

## 割り込み（interrupt）受信箱

`.work/skills/memory-bank/interrupt/` は **実装系作業の受信箱（funnel）**。非対話の放り込みも、対話中の
「作業を生む要望・指示」も、ここを必ず経由してから処理する＝**実装フローは interrupt を介す**（追跡すべき作業の
単一窓口・監査証跡）。interrupt を**バックログに保ち転記簿にしない**ため、下記は funnel しない。

- **非対話**: 別セッション・エディタ・ファイルコピー等で、思いついた要望・指示・メモ・バグ報告を**形式自由の
  ファイル**として直下に放り込んでおける。
- **対話**: ユーザーから対話で**実装系の作業を生む**要望・指示・方針を受けたら、エージェントは**着手前にまず
  それを interrupt にファイルとして書き出す**（生の入力を 1 ファイルに記録）。その上で下記の取り込みフロー
  （トリアージ→反映→archive）で処理する。

> **書き出さない（funnel 対象外）** — interrupt を「追跡すべき実装作業」のバックログに保つための線引き:
> 1. **問い合わせ・質問**（how / what / 状況確認 / コード説明 など読み取り専用で即答するもの）。
> 2. **コミット / プッシュ等のフロー操作**、および workflow を駆動する指示（`update memory bank` / 次サイクル /
>    interrupt 取り込み 等）。既存フローを回すだけで新規要件ではない。
> 3. **エージェントの質問への回答**（承認・却下・選択肢の決定）。既存作業への判断であって新規要件ではない。
> 4. **対話中の軌道修正・補足**で、すでに interrupt 化済み／作業中の項目を精緻化するもの → 新規ファイルを作らず
>    **当該タスク/項目に折り込む**（断片化を防ぐ）。
> 5. **雑談・あいさつ・相づち**。
>
> 迷ったら基準は「**後から独立に着手すべき実装作業が生まれるか**」。生まれるなら書き出す、回答・操作・会話なら書かない。

エージェントは **毎タスク開始時にここを必ず確認**し、中身を取り込んでから動く。

### 書き方（ユーザー側）

- `interrupt/` 直下に **任意のファイル**を置くだけ（ファイル名・拡張子・体裁は問わない。`.md` / `.txt` など）。
- 1 ファイルに 1 件でも複数件でもよい。粒度も問わない（「○○もやりたい」「××が壊れてる」程度でよい）。
- **順番を効かせたいときだけ**先頭に時刻（`YYYYMMDDHHMMSS-...`）を付けると取り込みが時系列になる（任意。
  人間は気にしなくてよい＝規約は強制しない）。
- 取り込み済みを退避する `interrupt/archive/` は読み飛ばす（後述）。

### 取り込み（エージェント側・起動時）

`interrupt/` 直下（`archive/` と `README.md` を除く）に未処理ファイルがあれば、コアワークフローに入る
前に次を行う:

1. **トリアージ** — 各ファイルの各項目について宛先を判定する:
   - 「いま/次にやる」具体作業 → **タスク化**（`TaskCreate`）＋ `activeContext.md`「現在のフォーカス」へ。
   - 将来の作業・要望 → `progress.md`「残作業」へ `M00x` として追記。
   - 方針・好み・知見 → `activeContext.md`「進行中の決定」/「重要なパターン・好み」や該当コアファイルへ。
   - 不明瞭・判断を要する → ユーザーに確認（破壊的・大物は特に）。
2. **反映** — 上記の宛先へ書き込む。何をどこへ振り分けたかを `activeContext.md`「直近の変更」に 1 行残す。
3. **クリア（取りこぼし防止）** — 取り込んだファイルは `interrupt/archive/` へ**日付プレフィックスを付けて移動**
   （例 `archive/2026-06-20-<元名>`。監査証跡＝消さずに退避）。直下からファイルが消えることで次回の
   二重取り込みを防ぐ。`archive/` が無ければ作る。

> 原則: **受信箱の内容を黙って捨てない**。取り込めない／保留する項目は archive へ動かさず `interrupt/` 直下に
> 残し、その旨をユーザーへ伝える。相対日付（「明日」等）は絶対日付に直して記録する。

### 取り込み順と優先度

- **取り込む順番＝ファイル名の辞書順**（`archive/`・`README.md` を除く直下を名前順に処理）。時刻プレフィックス
  が付いていれば自然に時系列になる。**規約は強制しない**——人間が適当な名前で放り込んだファイルもそのまま辞書順で扱う。
- **優先度は「取り込み順」ではなく「やる順」に効かせる**。ファイル内に `priority`（high/normal/low 等）があれば、
  それは backlog 化した後の **「現在のフォーカス」選定**で使う（取り込み自体は名前順で淡々と行う）。
- ワーカー文脈では、**要望された priority はあくまで要望**。実効優先度の確定は supervisor の判断
  （[[supervisor]] の「優先度の判断」）。supervisor が居ない standalone は常識的に自分で決める。

### 上りエスカレーション（worker → 親）

このプロジェクトが**ワーカーとして親の配下で動いている**場合（cwd が親プロジェクトの配下ディレクトリの中
＝構造から推測できる、または自分の宣言で親が分かる）に限り、「俯瞰判断・横断ルール・メタ層に関わる」気づきは
**抱え込まず親へ上げてよい**:

- 親の受信箱 `…/<親>/.work/skills/memory-bank/interrupt/` に 1 ファイル投函（書式は [[supervisor]] 準拠。
  `from` / `role: worker` / 出典 path:line・commit を添える）。非同期＝親の応答は待たず作業継続。
- 親が分からない／居ない（standalone）なら、エスカレせず自分の `progress.md`「残作業」に積む。
- 親へ手を出してよいのは**この interrupt 投函だけ**。親や他プロジェクトの中身は直接書き換えない。

## ドキュメント更新（Documentation Updates）

Memory Bank の更新は次のときに行う：
1. 新しいプロジェクトのパターンを発見したとき。
2. 重要な変更を実装したあと。
3. ユーザーが **「update memory bank」** を要求したとき（**全ファイルを必ずレビューする**）。
4. コンテキストの明確化が必要なとき。

更新プロセス: **全ファイルをレビュー → 現状を文書化 → 次のステップを明確化 → 知見・パターンを記録**。
特に **activeContext.md** と **progress.md**（現状を追う 2 ファイル）を重点的に見る。

## コミットをフローに組み込む（必須）

Memory Bank は引き継ぎの正本なので、**「切りのいいところ」で必ずコミットする**ところまでがワークフロー
（Act Mode の終端）。コミットしていない作業はリセットで失われる前提で扱う。

**前提：git リポジトリのときだけ。** リポジトリが git 初期化されていない場合（`git rev-parse` が失敗等）は
コミット手順を**まるごとスキップする**（`git init` を促したりコミットの是非を問うたりしない）。Memory Bank
ファイルの更新だけ行えばよい。

### 「切りのいいところ」の定義

次を満たしたら 1 つの区切り＝コミット単位とする：
1. 意味のあるまとまり（1 タスク / 1 機能 / 1 修正）が一段落した。
2. **品質チェックを行い**緑であること（何をどうチェックするかは**組織の品質方針**に従う。下記「品質チェックの所在」）。
   緑にできない場合は WIP である旨を activeContext.md に明記してからコミットする。
3. Memory Bank（特に **activeContext.md** と **progress.md**）を現状に合わせて更新済み。

> **品質チェックの所在（関心の分離）** — memory-bank は「品質チェックを行う」とだけ言い、**規約は持たない**。
> 一般的な検証方法・チェックシートは [[quality]] スキルが正本。**組織固有の適用**は `reference/`（参照系）の
> 品質方針エントリ（例 `reference/quality-policy.md` → [[quality]]）に置き、各 repo の `techContext.md` から参照する。

### コミットのやり方

- **コード変更と Memory Bank の更新を 1 コミットにまとめる**（「何を変えたか＋現状＋次の計画」が
  1 コミットで揃う）。
- **ブランチ方針は config で決める**（下記「コミット設定（config）」）。既定 `branchPolicy: agent-branch` は
  **エージェント専用ブランチ（既定名 `agent`）で作業・コミット**する。`current` なら現在ブランチへ直接、
  `new-branch` ならタスク毎に新ブランチ。
- メッセージは Conventional Commits（`feat:` / `fix:` / `refactor:` / `docs:` / `test:` 等）＋日本語要約で可。
- **push はユーザーが求めたときだけ**行う（コミットはこまめに、push は明示時）。
- コミット前に `git status` / `git diff` で意図しない混入（生成物・秘密情報）が無いか確認する。

### コミット設定（config）

ブランチ方針などは **config ファイルを読んで分岐**する（プロセに直書きしない）。コミット前に次の順で解決する:

1. **プロジェクト**: `.work/skills/memory-bank/config.json`（あれば最優先。repo と一緒に commit され方針が repo に同行）。
2. **グローバル既定**: スキル同梱の `<スキルのベースディレクトリ>/config.default.json`。
3. **ハードコード既定**: 上記が無い／キー欠落なら `branchPolicy = agent-branch`（`agentBranch = "agent"`）。

スキーマ（JSON）:

````json
{
  "commit": {
    "branchPolicy": "agent-branch",  // agent-branch=専用ブランチで作業 / current=現在ブランチへ直接 / new-branch=タスク毎に新規
    "agentBranch": "agent"           // branchPolicy=agent-branch のときの専用ブランチ名
  }
}
````

- **`agent-branch`（既定）の挙動**: 作業（最初の変更）に入る前に `agentBranch`（既定 `agent`）へ切り替える。
  無ければ現在の HEAD から作成（`git switch -c agent`）。既にそのブランチならそのまま。以後のコミットはここに
  単線で積む。切り替え時に未コミットの変更が競合する等の異常はユーザーに相談する。
- **理由**: per-task でブランチを乱立させるとマージが頻発し、多人数だと競合が起きてエージェントには確実な
  解決ができない（壊さず正しくマージできない）。そこで **専用ブランチ 1 本に単線でコミット**し、デフォルト
  ブランチへの**統合（マージ）は人間が行う**。エージェントはマージしない＝競合解決を背負わない。これで
  エージェントの作業を人間の `main` から隔離しつつ、競合の危険を避ける。
- **エージェントは `main` への統合を一切気にしない**。マージの提案・実行・進捗確認をせず、`agent` ブランチに
  専念してコミットを積むだけでよい（統合は人間に完全に委ねる）。
- 名前を変えたい／現在ブランチへ直接にしたい repo は `.work/skills/memory-bank/config.json` で
  `agentBranch` や `branchPolicy`（`current` / `new-branch`）を上書きする。
- config を増やすときも同じ解決順（project → global default → ハードコード）に従う。

> 自動化したい場合：判断（「切りがいいか」）を伴うため Stop フック等での無条件 auto-commit は避ける。
> どうしてもハードに強制したいなら、コミット漏れを*通知*するだけの非ブロッキングなフックに留める。

## 初期化（initialize memory bank）

ファイルが無いプロジェクトで初めて使うときの手順。**「提案して終わり」「通知して終わり」にせず、ユーザーが
承諾したら下記 1〜4 を最後（6 コアの書き出し）まで実行し切る**こと：

1. `mkdir -p .work/skills/memory-bank`。
2. `.gitignore` が `.work/` を無視していないか確認（無視していれば `!.work/skills/` 等で除外、または
   ユーザーに方針確認）。状態の正本なので **commit される**こと。
3. リポジトリを実際に調べ（README / pyproject / ソース / git log 等）、6 コアファイルを作成する。
   各ファイルの内容は上記「コアファイル」の役割に従い、**projectbrief.md（土台）→ 派生ファイル → 
   activeContext / progress** の順に埋める。下記「雛形」を出発点にしてよい。あわせて **`interrupt/`
   （受信箱ディレクトリ）を用意**する（中の `README.md` は任意。`archive/` は初回取り込み時に作ればよい）。
4. 初期化内容をユーザーに要約提示し、必要なら commit する。

## 注意

- **毎タスク開始時に全コアファイルを読む**（リセット後の唯一の手がかり）。これを省略しない。
- 大物タスクや破壊的・外向きの判断は着手前にユーザーへ確認する。
- リセット後は完全にゼロから始まる。Memory Bank だけが過去の作業との唯一の接続点であり、その正確さに
  作業効率が完全に依存する。**精確さと明瞭さをもって維持すること。**

### interrupt/（受信箱ディレクトリ・任意の拡張）

ディレクトリとして用意する。ユーザーは直下に任意のファイルを放り込む。雛形として置く `README.md` は任意:

```
interrupt/
├── README.md      # 任意（下記）。使い方メモ
└── archive/       # 取り込み済みの退避先（初回取り込み時に自動生成でよい）
```

````markdown
<!-- interrupt/README.md -->
# Interrupt（受信箱）

> このディレクトリ直下に、要望・指示・メモ・バグ報告を **任意のファイル**として放り込む（形式自由）。
> エージェントは毎タスク開始時に直下を取り込み、取り込んだファイルは archive/ へ日付つきで退避する。
> archive/ と本 README は取り込み対象外。
````
